# 圧縮ファイル対応 事前意思決定リスト

このドキュメントは、圧縮ファイルサポート実装前に合意しておくべき項目を整理したものです。

## 1. 決めることリスト

### 1. 対応スコープ
- MVP で対応する形式（例: ZIP のみ）
- 読み取り専用かどうか
- 非対応範囲（例: パスワード付き ZIP）

>zipのみ対応

>読み取り専用

なぜ必要か: 対応範囲が曖昧だと、設計とテストの見積もりが不安定になるため。

### 2. データソース識別子の仕様
- 通常フォルダと圧縮内部パスの表現方法
- 識別子の正規化ルール

>圧縮ファイルのパスに対応する一時解凍フォルダのパスを保持し、フロントエンドはそちらを参照する。利用者側が違いを意識する必要がないようにする

なぜ必要か: キャッシュキー、ナビゲーション、テストケースの一貫性に直結するため。

### 3. ナビゲーション仕様
- ZIP 内部の親フォルダ移動ルール
- ZIP を閉じて元のフォルダに戻る導線
- 同階層概念の扱い

なぜ必要か: 既存のフォルダ前提 UI と挙動差分が発生するため。

### 4. 取得アーキテクチャ
- 既存抽象の拡張か、新規サービス分離か
- SWR キー設計の拡張方針

なぜ必要か: 影響範囲と実装コストを最小化する設計分岐になるため。

### 5. 展開戦略
- 都度展開、ストリーミング、一時展開キャッシュの選択
- 容量上限、TTL、クリーンアップ方法

なぜ必要か: 大容量アーカイブ時の性能とディスク利用に直接影響するため。

### 6. セキュリティ仕様
- path traversal 対策
- 圧縮爆弾対策
- 許可拡張子と検証ポイント

なぜ必要か: ファイル処理機能では安全境界の明確化が最優先になるため。

### 7. サムネイルとキャッシュ無効化
- キャッシュキーの構成要素
- archive 更新時の無効化条件

なぜ必要か: 表示不整合や古いサムネイル残留を防ぐため。

### 8. 性能要件
- 初回表示の許容時間
- メモリ上限と並列度
- バックグラウンド処理方針

なぜ必要か: 実装後の性能評価基準がないと最適化判断ができないため。

### 9. エラー UX
- 破損アーカイブの表示
- 非対応形式のガイダンス
- 部分表示時のユーザー通知

なぜ必要か: 失敗時の体験品質が利用継続性に影響するため。

### 10. テスト方針
- 固定テストアーカイブの準備方法
- 回帰対象（フォルダ表示、サムネイル、ナビゲーション）
- E2E テスト観点

なぜ必要か: 圧縮対応は回帰範囲が広く、品質ゲートを先に固定する必要があるため。

### 11. フェーズ分割
- MVP（ZIP 閲覧）
- 形式拡張（7z, tar.gz など）
- 最適化フェーズ

なぜ必要か: 一括実装より段階的導入の方がリスク管理しやすいため。

## 2. 推奨方針（MVP）

1. ZIP のみ対応（MVP）
2. 読み取り専用（編集や再圧縮は対象外）
3. 既存抽象を維持して ZIP 用実装を追加
4. セキュリティ検証は Rust 側に集約
5. キャッシュ無効化は archive 更新検知を含める
