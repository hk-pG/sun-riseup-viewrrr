# Feature Specification: Rust Backend Thumbnail Optimization

**Feature Branch**: `001-rust-thumbnail-optimization`  
**Created**: 2026-01-05  
**Status**: Draft  
**Input**: User description: "Rust側でのサムネイル最適化処理を追加し、既存のフロントエンド最適化と比較評価する"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Fast Thumbnail Display During Scroll (Priority: P1)

ユーザーが大量の画像フォルダ（100件以上）を含むディレクトリを開いた際、サイドバーをスクロールしながら各フォルダのサムネイルを視認して目的のフォルダを見つける。

**Why this priority**: サムネイルビューアの中核となる機能であり、ユーザーが絵で判断しながら高速にフォルダを探索できることがアプリの価値の根幹。

**Independent Test**: 100枚の高解像度画像（各5MB、4000×3000px）を含むフォルダで、サイドバーを高速スクロール（1秒間に10〜20件のサムネイルが画面を通過）し、ユーザーが絵を認識しながら目的のフォルダを見つけられることを確認。

**Acceptance Scenarios**:

1. **Given** ユーザーが100個のフォルダを含むディレクトリを開いており、各フォルダには高解像度画像が含まれている, **When** サイドバーを素早くスクロールする, **Then** サムネイル画像が遅延なく表示され、スクロール動作がカクつかずに滑らかに動作する
2. **Given** ユーザーが高速スクロール中である, **When** スクロールを停止する, **Then** 表示されているサムネイルが即座に鮮明に表示される
3. **Given** ユーザーが一度スクロールしてサムネイルを表示済みである, **When** 同じ領域に再度スクロールバックする, **Then** サムネイルがキャッシュから即座に表示される

---

### User Story 2 - Initial Folder Load Performance (Priority: P2)

ユーザーが新しいディレクトリを開いた際、サイドバーに表示される全フォルダのサムネイルが適切な時間内で生成され表示される。

**Why this priority**: 初回ロード体験はアプリケーションの第一印象を決定づけ、ユーザーが待たされる時間を最小化する必要がある。

**Independent Test**: 新規ディレクトリ（100フォルダ）を開き、全サムネイルが表示されるまでの時間を計測し、ユーザーが待機中であることを示すUIが適切に機能することを確認。

**Acceptance Scenarios**:

1. **Given** ユーザーが新しいディレクトリを選択する, **When** ディレクトリが開かれる, **Then** 可視領域のサムネイルが2秒以内に表示される
2. **Given** サムネイル生成が進行中である, **When** ユーザーがUIを操作する, **Then** UIの応答性が維持され、生成処理がバックグラウンドで継続される
3. **Given** サムネイル生成が完了している, **When** 同じディレクトリを再度開く, **Then** キャッシュからサムネイルが即座に表示される（100ms以内）

---

### User Story 3 - Cross-Platform Performance Consistency (Priority: P3)

LinuxおよびmacOS環境において、サムネイル表示のパフォーマンスが同等レベルで動作する。

**Why this priority**: 既存実装ではLinux（WebKitGTK）環境で顕著なパフォーマンス劣化が発生しており、プラットフォーム間での体験の一貫性が求められる。

**Independent Test**: 同一のテストデータセット（100フォルダ、各5MB画像）を用いて、Linux（WebKitGTK）とmacOS（WKWebView）でスクロール性能とサムネイル表示速度を測定し、両者が同等の体験を提供することを確認。

**Acceptance Scenarios**:

1. **Given** ユーザーがLinux環境でアプリを使用している, **When** サイドバーを高速スクロールする, **Then** macOSと同等の滑らかさで動作する
2. **Given** ユーザーがLinux環境でディレクトリを開く, **When** サムネイルが生成される, **Then** macOSと同等の速度で表示される

---

### Edge Cases

- 非常に大きな画像ファイル（20MB以上、8000×6000px以上）のサムネイル生成
- サムネイル生成中にユーザーが別のディレクトリへ移動する
- ディスクキャッシュの容量が上限に達した場合のキャッシュクリーンアップ
- 破損した画像ファイルまたはサポート外のフォーマットの処理
- ネットワークドライブやリムーバブルメディア上の画像フォルダ

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、画像ファイルを指定されたサイズ（200×200px）にリサイズしたサムネイルを生成できなければならない
- **FR-002**: システムは、生成されたサムネイルをディスクキャッシュ（`$XDG_CACHE_HOME/sun-riseup-viewrrr/thumbnails/` または相当パス）に保存しなければならない
- **FR-003**: システムは、既にキャッシュされたサムネイルが存在する場合、再生成せずにキャッシュから読み込まなければならない
- **FR-004**: システムは、サムネイル生成を並列処理（最大4スレッド）で実行し、複数のサムネイルを同時に生成できなければならない
- **FR-005**: システムは、画像デコードエラーやファイル読み取りエラーが発生した場合でも、エラーを適切に処理し、デフォルトアイコンまたはプレースホルダーを表示しなければならない
- **FR-006**: システムは、サムネイルのファイル名として、元画像ファイルパスのハッシュ値を使用してキャッシュの一意性を保証しなければならない
- **FR-007**: ユーザーは、既存のフロントエンド最適化（スクロール中の画像描画スキップ）とRustバックエンド最適化を切り替えて比較できなければならない
- **FR-008**: システムは、サムネイル生成の進行状況をフロントエンドに通知し、ユーザーに適切なフィードバックを提供できなければならない

### Key Entities

- **Thumbnail**: 元画像から生成された200×200pxのリサイズ画像。キャッシュファイルとして保存され、メタデータ（元画像パス、生成日時、ファイルサイズ）を持つ。
- **ThumbnailCache**: サムネイルファイルを管理するキャッシュ機構。キャッシュの有効性チェック、古いキャッシュの削除、容量管理を担当。
- **GenerationTask**: サムネイル生成リクエスト。優先度、元画像パス、ターゲットサイズ、コールバック情報を含む。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーは、100個のフォルダを含むディレクトリで、サイドバーを1秒間に10〜20件のサムネイルが通過する速度でスクロールしても、カクつきなく滑らかに操作できる
- **SC-002**: 新規ディレクトリ（100フォルダ）を開いた際、可視領域の最初の10件のサムネイルが2秒以内に表示される
- **SC-003**: 既にキャッシュされたサムネイルは、100ms以内に表示される
- **SC-004**: サムネイル生成によるCPU使用率が、4コアマシンで平均50%以下に抑えられる
- **SC-005**: 既存のフロントエンド最適化とRustバックエンド最適化を実装したビルドを用意し、定量的なベンチマーク結果（スクロールFPS、初回ロード時間、メモリ使用量）を比較できる
- **SC-006**: Linux環境（WebKitGTK）において、macOS環境と比較してスクロール性能の差が10%以内に収まる

## Assumptions

- 対象画像フォーマットは JPEG, PNG, WebP とする（他のフォーマットは将来的な拡張として扱う）
- サムネイルの品質はJPEG品質60（バランス重視）とする
- キャッシュの最大容量は1GBとし、LRU（最も使われていないものから削除）方式でクリーンアップを行う
- 並列処理のスレッド数は環境のCPUコア数に基づいて動的に調整する（最小2、最大8）
- 元画像ファイルのタイムスタンプが変更された場合、キャッシュを無効化して再生成する

## Out of Scope

- 動画ファイルのサムネイル生成（将来的な拡張として別仕様で扱う）
- EXIF情報の抽出と表示（別機能として扱う）
- サムネイルのカスタムサイズ設定UI（現時点では200×200pxに固定）
- サムネイル生成のバッチ処理UI（全フォルダを事前生成する機能）
- クラウドストレージ上の画像の取り扱い

## Dependencies

- Rust `image` クレート（画像デコード・リサイズ処理）
- Rust `rayon` クレート（並列処理）
- Rust `sha2` または `blake3` クレート（キャッシュファイル名のハッシュ生成）
- Tauri のファイルシステムプラグイン（キャッシュディレクトリへのアクセス）
- 既存の `FileSystemService` インターフェースへの拡張（新しいコマンドの追加）

## Performance Benchmarks & Comparison Plan

この機能の目的の一つは、既存のフロントエンド最適化（スクロール中の画像描画スキップ）との定量的な比較です。以下の指標を測定し、両アプローチの長所・短所を明確化します。

### 測定指標

1. **スクロールパフォーマンス**
   - フレームレート（FPS）：高速スクロール時の平均FPS
   - フレームドロップ率：1秒あたりのフレーム落ち回数

2. **初回ロード時間**
   - 可視領域の最初の10件のサムネイルが表示されるまでの時間
   - 全100件のサムネイルが表示完了するまでの時間

3. **メモリ使用量**
   - アプリケーション全体のメモリ使用量（アイドル時、スクロール時、ピーク時）

4. **ディスク使用量**
   - キャッシュディレクトリのサイズ（100フォルダ分）

5. **CPU使用率**
   - サムネイル生成中の平均CPU使用率

### 比較方法

- 同一のテストデータセット（100フォルダ、各フォルダに5MB/4000×3000pxの画像）を使用
- Linux（WebKitGTK）とmacOS（WKWebView）の両環境で測定
- 各測定を3回実行し、平均値を記録
- 結果を表形式でドキュメント化し、core ブランチへの統合判断材料とする
