# Feature Specification: サイドバーとビューアのスクロール動作修正

**Feature Branch**: `003-fix-sidebar-scroll`  
**Created**: 2026年1月3日  
**Status**: Draft  
**Input**: ユーザー説明: "サイドバーとビューア部分の高さが動悸してしまい、不要な縦スクロールが発生する。一般的なビューアソフトと同じようにビューアサイドバーはビューア部分の高さに合わせてそれ以降はスクロールするようにしたい。逆にビューア部分は基本的にスクロールはしないようにしたい（ズームしない限り）"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ビューア画像の全体表示 (Priority: P1)

ユーザーが画像ビューアで画像を開いたとき、画像全体がビューアエリア内に収まって表示され、ビューアエリア自体に不要なスクロールバーが表示されない。

**Why this priority**: 画像ビューアの最も基本的な機能であり、ユーザーが画像を快適に閲覧するための必須要件。現在の問題（不要な縦スクロール）が最も影響を与えている部分。

**Independent Test**: 任意の画像を開き、ビューアエリアにスクロールバーが表示されないこと、画像全体が見えることを確認することで、単独でテスト可能。

**Acceptance Scenarios**:

1. **Given** ユーザーがフォルダを開いて画像を選択している状態、**When** 画像ビューアに画像が表示される、**Then** ビューアエリアには縦スクロールバーが表示されず、画像全体がビューアエリア内に収まって表示される
2. **Given** ビューアに画像が表示されている状態、**When** ウィンドウサイズを変更する、**Then** 画像は新しいビューアエリアサイズに合わせて再スケールされ、スクロールバーは表示されない
3. **Given** ビューアに画像が表示されている状態、**When** 異なるサイズの画像に切り替える、**Then** 新しい画像もビューアエリア内に収まり、スクロールバーは表示されない

---

### User Story 2 - サイドバーの独立スクロール (Priority: P1)

フォルダ一覧やサムネイル一覧がサイドバーに多数表示されている場合、サイドバーのみが独立してスクロール可能であり、ビューアエリアの表示には影響しない。

**Why this priority**: フォルダやファイルが多い場合の操作性に直結する重要な機能。一般的な画像ビューアやファイルマネージャーの標準的な動作。

**Independent Test**: 多数のフォルダまたは画像があるディレクトリを開き、サイドバーをスクロールしてビューアが固定されたままであることを確認することで、単独でテスト可能。

**Acceptance Scenarios**:

1. **Given** サイドバーに表示しきれない数のフォルダ/ファイルがある状態、**When** サイドバー内でスクロールする、**Then** サイドバーの内容のみがスクロールし、ビューアエリアの画像表示は固定されたまま
2. **Given** サイドバーをスクロールしている状態、**When** ビューアエリアの画像を操作する（次の画像に移動など）、**Then** サイドバーのスクロール位置は保持されたまま、画像のみが切り替わる
3. **Given** ビューアエリアに画像が表示されている状態、**When** サイドバーを最下部までスクロールする、**Then** ビューアエリアの画像表示は変わらず、サイドバー内のみでスクロールが完結する

---

### User Story 3 - ズーム時のビューアスクロール (Priority: P2)

ユーザーが画像をズームイン（拡大）したとき、画像がビューアエリアより大きくなった場合のみ、ビューアエリアにスクロールバーが表示され、画像の隠れた部分を見ることができる。

**Why this priority**: 画像の詳細を確認したいときの必須機能だが、通常の閲覧（P1）ほど頻繁には使用されない。

**Independent Test**: 任意の画像を開いてズーム機能を使用し、ズーム時のみスクロールバーが表示されることを確認することで、単独でテスト可能。

**Acceptance Scenarios**:

1. **Given** ビューアに通常サイズの画像が表示されている状態、**When** ユーザーが画像をズームイン（拡大）する、**Then** 画像がビューアエリアより大きくなった時点でスクロールバーが表示され、画像全体をスクロールして閲覧できる
2. **Given** ズームインされてスクロールバーが表示されている状態、**When** ユーザーがズームアウト（縮小）して画像がビューアエリア内に収まる、**Then** スクロールバーが非表示になり、画像全体が表示される
3. **Given** ズームインされた画像をスクロールして一部を表示している状態、**When** ユーザーがズームアウトする、**Then** スクロール位置がリセットされ、画像全体が中央に配置される

---

### Edge Cases

- ウィンドウサイズが極端に小さい場合（最小ウィンドウサイズ）でも、サイドバーとビューアが適切に表示されるか？
- サイドバーが空の状態（フォルダ/ファイルが0件）でも、レイアウトが崩れないか？
- 縦長画像と横長画像を切り替えたとき、レイアウトが適切に調整されるか？
- ズーム操作を高速に繰り返したとき、スクロールバーの表示/非表示が正しく切り替わるか？
- 画像を表示中にサイドバーの幅を変更（リサイズ）したとき、ビューアエリアが適切に再計算されるか？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、サイドバーとビューアエリアを独立した縦方向のスクロール領域として管理しなければならない
- **FR-002**: ビューアエリアは、画像がズームされていない状態では縦方向のスクロールバーを表示してはならない
- **FR-003**: サイドバーは、表示すべきコンテンツがサイドバーの高さを超える場合、独立した縦スクロールバーを表示しなければならない
- **FR-004**: ビューアエリアは、画像がズームインによってビューアエリアのサイズを超えた場合のみ、スクロールバーを表示しなければならない
- **FR-005**: サイドバーのスクロール操作は、ビューアエリアの表示状態に影響を与えてはならない
- **FR-006**: ビューアエリアの高さは、アプリケーションウィンドウの高さからヘッダー（メニューバー等）の高さを引いた値と一致しなければならない
- **FR-007**: サイドバーの高さは、ビューアエリアの高さと同じでなければならない
- **FR-008**: ウィンドウサイズが変更されたとき、サイドバーとビューアエリアの高さは動的に再計算されなければならない
- **FR-009**: 画像切り替え時、ビューアエリアのスクロール位置はリセット（トップに戻る）されなければならない
- **FR-010**: ズームアウトによって画像がビューアエリア内に収まる場合、ビューアエリアのスクロールバーは非表示にならなければならない

### Key Entities

- **ビューアエリア**: 画像を表示する主要な領域。高さはウィンドウ高さから固定要素（ヘッダー等）を引いた値。通常はスクロールなし、ズーム時のみスクロール可能。
- **サイドバー**: フォルダ一覧やファイル一覧を表示する領域。ビューアエリアと同じ高さを持ち、コンテンツが多い場合は独立してスクロール可能。
- **画像コンテンツ**: ビューアエリア内に表示される画像。通常はビューアエリア内に収まるようにスケールされ、ズーム時のみビューアエリアより大きくなる可能性がある。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーが画像を通常表示（ズームなし）で閲覧している間、ビューアエリアに縦スクロールバーが表示されない（100%のケースで達成）
- **SC-002**: サイドバーに10個以上のアイテムが表示されている状態で、サイドバーのスクロール操作がビューアエリアの表示に影響を与えない（100%のケースで達成）
- **SC-003**: ウィンドウサイズの変更後も、サイドバーとビューアエリアの高さが同期し、不要なスクロールバーが表示されない（100%のケースで達成）
- **SC-004**: 画像をズームインしてビューアエリアを超えたとき、ビューアエリアにスクロールバーが表示され、画像の全範囲をスクロールして閲覧できる（100%のケースで達成）
- **SC-005**: 異なるサイズの画像（縦長、横長、正方形）を切り替えたとき、各画像が適切にビューアエリア内に収まり、スクロールバーの表示/非表示が正しく制御される（100%のケースで達成）

## Assumptions

1. **レイアウトモデル**: アプリケーションは固定ヘッダー（メニューバー）と、その下に横並びのサイドバー+ビューアエリアを持つレイアウトを採用していると仮定
2. **ズーム機能**: 既存のズーム機能（マウスホイール、ボタン等）が実装されており、その機能は維持されると仮定
3. **CSS/スタイリング**: Tailwind CSS 4を使用しており、flexboxまたはgridレイアウトによる実装が可能と仮定
4. **画像スケーリング**: 画像は通常、ビューアエリアに収まるように自動スケールされる現在の動作を維持すると仮定
5. **ブラウザ互換性**: Tauri v2がサポートするWebレンダリングエンジン（WebView2等）で動作すれば十分と仮定

## Scope

### In Scope

- サイドバーとビューアエリアの高さ同期の実装
- サイドバーの独立スクロール機能の実装
- ビューアエリアのスクロール制御（通常時は無効、ズーム時のみ有効）
- ウィンドウリサイズ時の動的な高さ再計算
- 既存のレイアウトコンポーネント（AppShell、FolderNavigation、ImageViewer）の修正

### Out of Scope

- 新しいズーム機能の追加（既存のズーム機能の改善は対象外）
- サイドバーの幅変更（リサイズハンドル）機能の追加
- 水平スクロールの制御（縦スクロールのみが対象）
- サイドバーの表示/非表示トグル機能
- 新しいキーボードショートカットの追加
- パフォーマンス最適化（スクロール動作の改善のみに焦点）
